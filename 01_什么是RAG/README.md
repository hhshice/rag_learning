# ① 什么是 RAG（Retrieval-Augmented Generation）

## 🧠 概念解释

### RAG 解决什么问题？

大语言模型（如 ChatGPT）有两个致命弱点：

1. **知识过时** — 它只记得训练数据截止日期之前的事，训练之后发生的事它不知道
2. **会"幻觉"** — 它不知道答案的时候，会一本正经地胡说八道

**RAG 的核心思路：让大模型"开卷考试"**

> 普通对话 = 闭卷考试 → 只能靠记忆，记不住就编
> RAG 对话 = 开卷考试 → 先从资料库里找到相关内容，再根据找到的内容来回答

### RAG 的工作流程（三步走）

```
用户提问 → [1. 检索相关资料] → [2. 把资料 + 问题一起喂给大模型] → [3. 大模型生成回答]
```

用现实生活中的例子类比：

| 步骤 | 类比 | RAG 中的对应 |
|------|------|-------------|
| 1 | 去图书馆查资料 | 从向量数据库中检索相关文档片段 |
| 2 | 把资料摊在桌上，看着资料回答 | 把检索结果拼入 Prompt |
| 3 | 根据资料写出答案 | 大模型基于上下文生成回答 |

### RAG vs 微调 vs 纯 Prompt

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 纯 Prompt | 简单 | 无法使用私有数据 | 通用问答 |
| 微调（Fine-tuning） | 模型"记住"知识 | 成本高、知识难更新、容易忘 | 改变模型风格/语气 |
| **RAG** | 知识可随时更新、成本低、可溯源 | 依赖检索质量 | **企业知识库、客服、文档问答** |

**关键结论：RAG 是当前最主流的让大模型拥有"私有知识"的方式。**

## 📦 示例代码

下面这个示例展示了 RAG 的核心逻辑（不涉及向量数据库，用简单的字符串匹配代替，帮你建立直觉）：

```python
# mini_rag_demo.py

# ===== 第一步：准备知识库 =====
knowledge_base = [
    "RAG 全称是 Retrieval-Augmented Generation，即检索增强生成。",
    "RAG 由 Facebook 在 2020 年提出，核心思想是让大模型在生成回答前先检索相关文档。",
    "RAG 的主要优势是可以避免大模型产生幻觉，同时知识可以实时更新。",
    "向量数据库是 RAG 系统的核心组件之一，用于存储和检索文档的向量表示。",
    "Embedding 是将文本转换为向量的技术，是 RAG 的基础。",
]

# ===== 第二步：检索（简化版，用关键词匹配代替向量检索） =====
def simple_search(query, documents, top_k=3):
    """最简单的检索：计算 query 和每个文档的重叠词数量"""
    query_words = set(query.lower().split())
    scored = []
    for doc in documents:
        doc_words = set(doc.lower().split())
        score = len(query_words & doc_words)  # 交集大小作为相似度
        scored.append((score, doc))
    scored.sort(reverse=True)
    return [doc for score, doc in scored[:top_k]]

# ===== 第三步：构建 Prompt 并调用大模型 =====
def build_prompt(query, retrieved_docs):
    prompt = f"""你是一个问答助手。请根据以下参考资料回答用户问题。
如果参考资料中没有相关信息，请如实说明。

参考资料：
{chr(10).join(f'- {doc}' for doc in retrieved_docs)}

用户问题：{query}

请根据参考资料回答："""
    return prompt

# ===== 完整流程 =====
if __name__ == "__main__":
    user_query = "RAG 是什么？它有什么好处？"

    # 检索
    retrieved = simple_search(user_query, knowledge_base, top_k=3)
    print("=" * 50)
    print("检索到的相关文档：")
    for i, doc in enumerate(retrieved, 1):
        print(f"  [{i}] {doc}")

    # 构建 Prompt
    prompt = build_prompt(user_query, retrieved)
    print("\n" + "=" * 50)
    print("构建的 Prompt：")
    print(prompt)

    print("\n" + "=" * 50)
    print("在真实项目中，上面的 Prompt 会被发送给大模型 API 来获取最终回答。")
```


## ⚠️ 常见坑

1. **把 RAG 当成万能药** — RAG 只能解决"知识不足"的问题，不能解决模型推理能力差的问题
2. **检索质量决定一切** — 如果检索不到相关文档，大模型再强也答不对。RAG 系统的核心难点在"检索"而不是"生成"
3. **混淆 RAG 和微调** — 微调是让模型"记住"知识（写进权重），RAG 是让模型"查阅"知识（写进数据库）。两者互补，不是替代关系

---

## 🚀 实战建议

1. **先理解"为什么"再学"怎么做"** — 现阶段只需要知道 RAG = 检索 + 生成，不需要深究技术细节
2. **关注完整链路** — RAG 不是单一技术，而是一条流水线：文档加载 → 切分 → 向量化 → 存储 → 检索 → 排序 → 生成。后续会一步步拆解
3. **实际应用场景举例**：
   - 企业内部知识库问答（员工问制度、流程）
   - 客服智能助手（基于产品手册回答客户问题）
   - 法律/医疗文档分析（基于专业文献回答问题）
   - 代码助手（基于代码仓库回答编程问题）
